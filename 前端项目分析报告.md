## 一、严重程度【高】的问题（必须修复）

### H1. 安全性：登录鉴权完全形同虚设
- **文件**：`src/stores/user.js` 第 11-22 行
- **文件**：`src/views/LoginView.vue` 第 59-71 行
- **问题描述**：
  - `login()` 函数只要 `name` 和 `pwd` 非空就视为登录成功，完全不校验密码
  - token 为 `'mock-token-' + Date.now()`，纯前端伪造
  - 任何用户名输入任意密码都能登录
  - LoginView 第 68 行错误提示文案"用户名或密码不能为空"与实际逻辑不符（实际是任意非空都通过）
- **改进建议**：接入后端鉴权接口，密码至少做前端长度+格式校验，token 由后端下发，前端只存储

### H2. 安全性：路由守卫仅靠 token 存在性判断
- **文件**：`src/router/index.js` 第 38-55 行
- **问题描述**：`isLoggedIn()` 仅判断 `!!token.value`，任何人在 localStorage 写入任意 token 即可绕过登录
- **改进建议**：增加 token 有效性校验（过期时间、签名验证），首次进入页面时调用 `/me` 接口验证

### H3. 性能：Element Plus 与图标全量注册，首屏体积过大
- **文件**：`src/main.js` 第 3-14 行
- **问题描述**：
  - `import ElementPlus from 'element-plus'` 全量引入所有组件
  - `for (const [key, component] of Object.entries(ElementPlusIconsVue))` 全量注册所有图标（287 个）
  - 实际项目仅用到不到 20 个图标
- **改进建议**：使用 `unplugin-vue-components` + `unplugin-auto-import` 按需引入组件与图标，预计可减少 60%+ 首屏体积

### H4. 健壮性：localStorage 解析无 try-catch，损坏数据会导致白屏
- **文件**：`src/stores/data.js` 第 109 行
- **问题描述**：`JSON.parse(localStorage.getItem('alarms') || 'null')` 如果 localStorage 中存了非法 JSON 字符串，整个 store 初始化崩溃，应用白屏
- **改进建议**：包裹 try-catch，解析失败时回退到默认数据并清理脏数据

### H5. 健壮性：DeviceManageView 空值访问会崩溃
- **文件**：`src/views/DeviceManageView.vue` 第 86 行
- **问题描述**：`store.units.find(u => u.id === store.selectedDevice?.split('-')[0])?.name` —— `selectedDevice` 初始值可能为 `null`（store 第 275 行 `selectedDevice = ref('U1-idf-a')`，但删除设备后第 156 行设为 `null`），此时 `null?.split` 返回 `undefined`，`find(u => u.id === undefined)` 没问题，但 `unitName` 计算属性在 `selectedDevice` 为 `null` 时仍会触发，需确认安全
- **改进建议**：增加显式判空 `if (!store.selectedDevice) return '-'`

### H6. 安全性：使用 innerHTML 渲染动态内容，存在 XSS 风险
- **文件**：`src/views/DashboardView.vue` 第 219、222、256-271 行
- **文件**：`src/views/DeviceManageView.vue` 第 55、114 行
- **文件**：`src/views/TraceView.vue` 第 24、34 行（`v-html="s.body"` / `v-html="curCase.conclusion"`）
- **问题描述**：
  - DashboardView 用 `document.getElementById('dash-insight').innerHTML = text` 直接操作 DOM，绕过 Vue 响应式系统
  - 多处 `v-html` 渲染包含用户输入或动态生成的内容
  - 设备名、报警描述等用户可编辑内容若包含 `<script>` 或恶意 HTML，会被执行
- **改进建议**：改为 Vue 响应式数据绑定；如必须用 v-html，需先做 DOMPurify 过滤

### H7. 健壮性：编辑规则时"先删后加"无事务保护
- **文件**：`src/views/AlarmConfigView.vue` 第 313-329 行
- **问题描述**：编辑保存时先 `store.delRule(form.value.id)` 再循环 `store.addRule()`，如果中途某条 addRule 抛错，原规则已删除，新规则未完整添加，数据丢失
- **改进建议**：先构建完整新规则数组，校验通过后一次性替换；或使用 store 的 update 方法

### H8. 功能缺陷：模拟新报警按钮实际不触发报警
- **文件**：`src/views/AlarmListView.vue` 第 108-118 行
- **问题描述**：`showSimulate` 函数弹出 prompt 后仅 `ElMessage.info('已演示模式：实际触发应从设备选择器选择')`，没有调用 `store.triggerAlarm`，按钮形同虚设
- **改进建议**：使用 `el-select` 让用户选择设备，调用 `store.triggerAlarm(deviceId, level, desc, point, val)` 真实触发

### H9. 性能：ECharts 实例未监听窗口 resize，缩放后图表变形
- **文件**：`src/views/DashboardView.vue` 第 156-185 行
- **文件**：`src/views/GlobalAlertView.vue` 第 166-194 行
- **文件**：`src/views/ConditionAnalysisView.vue` 第 80-107 行
- **文件**：`src/views/StatsView.vue` 第 108-193 行
- **文件**：`src/views/AgentView.vue` 第 256-310 行
- **问题描述**：所有 ECharts 实例都没有 `window.addEventListener('resize', ...)`，浏览器窗口缩放后图表不会自适应
- **改进建议**：封装 `useECharts` composable，统一处理 init/setOption/resize/dispose

### H10. 性能：DashboardView setInterval 状态泄漏风险
- **文件**：`src/views/DashboardView.vue` 第 287-288 行
- **问题描述**：`iv = setInterval(updateMetrics, 3000)` 在 setup 顶层执行（非 onMounted 内），如果组件被 keep-alive 缓存再激活，interval 不会被重建；onUnmounted 清理时 `c1?.dispose()` 可能因 c1 未初始化报错
- **改进建议**：将 setInterval 移入 onMounted，clearInterval 移入 onUnmounted，增加初始化状态判断

### H11. 导航结构不合理：报警诊断归类错误
- **文件**：`src/components/AppLayout.vue` 第 16-36 行
- **问题描述**：
  - "报警诊断"放在"监控中心"分组下，但它属于智能分析功能
  - "故障溯源"在"智能分析"分组，但与"报警诊断"功能高度相关，二者分散
  - "统计报表"放在"设备与配置"下，但它是分析类功能
  - 三个分组分类逻辑不清晰
- **改进建议**：重组为「实时监控」（监盘/大屏/报警/工况）+「配置管理」（设备/报警配置/分级）+「智能分析」（诊断/溯源/AI/知识图谱/统计）

### H12. 可用性：批量操作无二次确认，误操作不可逆
- **文件**：`src/views/AlarmListView.vue` 第 97-106 行
- **问题描述**：`batchConfirm` 和 `batchResolve` 一键将所有未处理/已确认报警批量改状态，无 `ElMessageBox.confirm` 二次确认，值班员误点会导致大量报警被错误处理
- **改进建议**：增加二次确认弹窗，显示具体影响的报警数量

## 二、严重程度【中】的问题

### M1. 交互：AgentView 全屏后无返回入口
- **文件**：`src/components/AppLayout.vue` 第 3、41、50 行；`AgentView.vue` 第 469 行
- **问题描述**：进入 `/agent` 后 `isAgentPage=true`，侧栏和顶栏全部隐藏（`v-if="!isAgentPage"`），AgentView 自定义顶栏只有"在线/用户名"，没有返回主界面的入口，用户只能手动改 URL
- **改进建议**：在 AgentView 顶栏左侧加"返回"按钮，调用 `router.push('/dashboard')`

### M2. 交互：所有列表均无分页
- **文件**：`src/views/AlarmListView.vue` 第 20 行
- **文件**：`src/views/AlarmConfigView.vue` 第 18-35 行（虚拟列表，但无分页）
- **问题描述**：报警列表全量渲染所有报警，生产环境报警可能上千条，会导致卡顿
- **改进建议**：使用 el-pagination 分页或 el-table 虚拟滚动

### M3. 交互：表格无空状态、无 loading 状态
- **文件**：`src/views/AlarmListView.vue` 第 20 行
- **文件**：`src/views/DeviceManageView.vue` 第 39 行
- **文件**：`src/views/AlarmGradeView.vue` 第 22-30 行
- **问题描述**：所有 el-table 都没有 `empty-text` 自定义、没有 `v-loading` 指令，数据加载时无反馈，数据为空时只有默认"暂无数据"
- **改进建议**：统一封装带 loading + 自定义空状态的表格组件

### M4. 交互：AlarmConfigView 表单验证不完善
- **文件**：`src/views/AlarmConfigView.vue` 第 305-311 行
- **问题描述**：
  - 未使用 `el-form` 的 `ref` + `validate` 方法
  - 验证逻辑分散在 save 函数中，只校验了 device/unit/rules.val，未校验 pointKey、alarmDesc 等必填项
  - 表单顶部声明"带 * 为必填项"，但实际未强制校验
- **改进建议**：使用 `:rules` + `formRef.validate()` 统一校验

### M5. 数据真实性：avgTime/totalTimes 等用 Math.random 模拟，每次刷新都变
- **文件**：`src/views/AlarmGradeView.vue` 第 61-66 行
- **文件**：`src/views/StatsView.vue` 第 65-67 行
- **文件**：`src/views/TraceView.vue` 第 83、87-88 行
- **问题描述**：`avgTime` 返回 `(Math.random() * 2 + store.alarmLevels[lv].avgTime * 0.5).toFixed(1)`，每次访问 computed 都重新计算，数值跳动；StatsView 的 `filterCnt/totalTimes/totalMins` 同样
- **改进建议**：模拟数据应在 onMounted 时生成一次并存入 ref，或基于实际报警数据计算

### M6. 功能缺陷：StatsView 切换 today/week/month 不切换数据
- **文件**：`src/views/StatsView.vue` 第 3-7、195 行
- **问题描述**：tabs 切换 period 后只触发 `initCharts` 重新绘制，但所有数据源（stats、profCnt、cnt）都是基于 `store.alarms` 全量计算的，与 period 无关，用户切换 tab 看到的是相同数据
- **改进建议**：根据 period 过滤报警时间，或调用后端 `statsApi.overview(range)`

### M7. 语法/逻辑：ConditionAnalysisView 同一元素两个 style 属性
- **文件**：`src/views/ConditionAnalysisView.vue` 第 13 行
- **问题描述**：`<div class="mv" :style="{color:cur.health>=80?'#22c55e':'#f59e0b'}" style="font-size:18px">` —— 同时有 `:style` 和 `style`，Vue 会合并但可读性差，且 `:style` 会覆盖 `style` 中相同属性
- **改进建议**：合并为一个 `:style` 对象

### M8. 语法缺陷：AgentView 对象字面量重复键
- **文件**：`src/views/AgentView.vue` 第 200 行、第 453 行、第 456 行、第 459 行
- **问题描述**：`{ id: 'chart-1', source: 'Surface', id: 's5_b8391b357e7', title: '引风机工况分析', toggle: true }` —— `id` 键重复，后值覆盖前值，会导致 `chart-1` 这个 id 丢失，`renderChart` 中的 `if (id === 'chart-1')` 判断永远不成立
- **改进建议**：将第二个 id 改名为 `sourceId` 或 `tagId`

### M9. 交互：删除设备按钮未标红，与删除机组不一致
- **文件**：`src/views/DeviceManageView.vue` 第 25 行
- **问题描述**：`<el-button size="small" type="danger" @click="delCurDev">` 用了 type="danger"，但删除机组的 `.tu-del` 仅 hover 变红，两处危险操作的视觉提示不一致
- **改进建议**：统一危险操作的视觉规范

### M10. 交互：AgentView 无删除/清空历史对话功能
- **文件**：`src/views/AgentView.vue` 第 27-35 行
- **问题描述**：历史对话列表只能点击切换，不能删除单条、不能清空全部，长期使用后列表会无限增长
- **改进建议**：每条对话加删除按钮（hover 显示），顶部加"清空全部"

### M11. 交互：AgentView send 函数无 loading、无网络请求
- **文件**：`src/views/AgentView.vue` 第 320-343 行
- **问题描述**：
  - `send` 仅用 `setTimeout(600)` 模拟回复，无真实 API 调用
  - 等待回复期间无 loading 指示，用户不知道是否在处理
  - 输入框未禁用，用户可连续发送
- **改进建议**：增加 `sending` 状态，输入框 disabled，显示打字指示器

### M12. 交互：GlobalAlertView 大屏缺少全屏/返回入口
- **文件**：`src/views/GlobalAlertView.vue` 整个文件
- **问题描述**：作为"大屏"页面，缺少 F11 全屏按钮、缺少返回常规布局的入口；侧栏被隐藏（因为还在 AppLayout 内，但侧栏可见），与"大屏"定位不符
- **改进建议**：参考 AgentView 全屏化处理，增加全屏切换按钮

### M13. 健壮性：DeviceManageView watch 导入位置不规范
- **文件**：`src/views/DeviceManageView.vue` 第 101 行
- **问题描述**：`import { watch } from 'vue'` 写在代码中间，而非文件顶部，可读性差
- **改进建议**：合并到顶部 import 语句

### M14. 一致性：报警状态文字转换函数三处重复定义且不一致
- **文件**：`AlarmListView.vue` 第 87 行、`GlobalAlertView.vue` 第 125 行、`DashboardView.vue` 第 87 行（隐式）

### M16. 一致性：卡片样式 cd/cd-t/cd-h 在 6+ 文件中重复定义
- **文件**：`DashboardView.vue`、`GlobalAlertView.vue`、`AlarmListView.vue`、`AlarmGradeView.vue`、`DiagnosisView.vue`、`ConditionAnalysisView.vue`、`StatsView.vue`、`DeviceManageView.vue`、`TraceView.vue`、`KnowledgeGraphView.vue` 均有

### M17. 性能：DashboardView.tsArr 在 setup 顶层创建，组件复用会共享
- **文件**：`src/views/DashboardView.vue` 第 152-154 行
- **问题描述**：`const tsArr = Array.from(...)` 在 setup 顶层（模块级），如果组件被多次实例化，所有实例共享同一数组
- **改进建议**：移入 setup 函数内部或用 computed 包裹

### M18. 可访问性：全局缺少 aria 属性，键盘导航支持不足
- **文件**：所有 `.vue` 文件
- **问题描述**：

  - 屏幕阅读器无法识别

### M19. 响应式：除 LoginView 外无任何响应式适配
- **文件**：`AppLayout.vue`、`DashboardView.vue`、`GlobalAlertView.vue` 等
- **问题描述**：
  - 仅 `LoginView.vue` 第 216 行有 `@media (max-width: 480px)`

  - AppLayout 侧栏固定 220px，移动端无折叠
- **改进建议**：增加 `lg/md/sm/xs` 响应式断点，侧栏增加折叠功能

### M20. 对比度：多处文字颜色与背景对比度不足
- **文件**：`global.css` 第 17 行（`.cd-t color: #94a3b8`）、多处 `color: #64748b`

- **改进建议**：次要文字提亮至 `#94a3b8`（对比度 5.4:1），placeholder 提亮至 `#64748b`

### M21. 对比度：ECharts 图表文字过小
- **文件**：`DashboardView.vue` 第 161 行 `fontSize: 8`、第 162 行 `fontSize: 10`
- **文件**：`GlobalAlertView.vue` 第 186 行 `fontSize: 9`
- **问题描述**：图表坐标轴文字 8-10px，在 1920×1080 屏幕上几乎不可读，工业现场大屏更严重
- **改进建议**：坐标轴文字至少 12px，图例 13px

### M22. 交互：登录页密码框未禁止复制/粘贴，无验证码
- **文件**：`src/views/LoginView.vue` 第 22 行
- **问题描述**：工业系统登录无验证码、无密码强度提示、无登录失败次数限制
- **改进建议**：增加图形验证码，连续失败 5 次锁定

## 三、严重程度【低】的问题

### L1. 命名一致性：变量命名风格混乱
- **文件**：`stores/data.js` 中报警对象用 `l`（level）、`st`（status）、`desc` 等单字母/缩写
- **文件**：`DashboardView.vue` 用 `c1/c2/c3`（chart1）、`iv`（interval）
- **文件**：`AgentView.vue` 用 `fns`、`qq`、`x`（query）
- **问题描述**：中英混用、缩写不统一、单字母变量无注释
- **改进建议**：建立项目命名规范，level/status 等用全拼

### L2. 注释不足：核心业务逻辑缺少注释
- **文件**：`stores/data.js` 第 99-105 行 `updateDeviceParam` 健康度计算公式 `d.health * 0.4 + 60 * (ok ? 0.95 : 0.5)` 无注释
- **文件**：`ReasoningGraph.vue` 第 101 行 `yLayer = [28, 95, 175, 245, 320]` 魔法数字无注释
- **改进建议**：关键算法补充注释说明

### L3. 状态色规范：橙色 #f59e0b 在深色背景上辨识度偏低
- **文件**：多处使用
- **问题描述**：`#f59e0b`（橙）与 `#fbbf24`（中黄）在 `#111827` 背景上差异不明显，"二级报警"和"中间现象"容易混淆
- **改进建议**：参考工业标准 ISA 101，预警用更饱和的橙红 `#fb923c`

### L4. 图标可识别性：使用 emoji 替代图标，跨平台显示不一致
- **文件**：`DashboardView.vue` 第 6、35-37 行（`⚡🔥⚙`）、`GlobalAlertView.vue` 第 68、83 行（`⚠️🤖`）、`AgentView.vue` 第 160-168 行（`📊🔔📈🔍📚📋🎯📝`）
- **问题描述**：emoji 在 Windows/Mac/Linux 浏览器上渲染样式不同，工业环境可能显示为方框
- **改进建议**：统一使用 Element Plus Icons 或 SVG 图标

### L5. 顶栏信息不完整：缺少机组切换、通知中心、用户头像
- **文件**：`src/components/AppLayout.vue` 第 41-49 行
- **问题描述**：顶栏只有标题/用户名/状态/时钟/退出，缺少：
  - 全局机组切换器（目前需进入 Dashboard 才能切换）
  - 通知中心入口（报警推送）
  - 用户头像下拉菜单（个人设置/修改密码）
  - 主题切换
- **改进建议**：参考工业 SCADA 系统顶栏规范

### L6. 面包屑缺失：仅显示当前页面标题，无层级路径
- **文件**：`src/components/AppLayout.vue` 第 42 行
- **问题描述**：`<span class="title">{{ $route.meta?.title || '' }}</span>` 仅显示一级标题，无"监控中心 > 智能监盘"这样的层级路径
- **改进建议**：使用 el-breadcrumb，结合路由 meta 生成层级

### L7. 菜单选中态：依赖 route.path 完整匹配，子路由不高亮
- **文件**：`src/components/AppLayout.vue` 第 9 行
- **问题描述**：`:default-active="$route.path"` 在路由变化时不会响应式更新（default-active 只在初始化生效），且如果未来有子路由（如 `/alarms/detail/123`），菜单不会高亮
- **改进建议**：使用 `:default-active="route.path"` 并加 `watch`，或改用 `router-link` 的 active class

### L8. 代码重复：alarmLevels 等大对象定义在 store 内
- **文件**：`src/stores/data.js` 第 173-212 行
- **问题描述**：alarmLevels、kgData、kgRelations 等静态数据混在 store 中，store 文件过长（317 行），可维护性差
- **改进建议**：拆分到 `stores/data/alarmLevels.js`、`stores/data/kg.js` 等独立文件

### L9. 性能：DiagnosisView 硬编码诊断数据
- **文件**：`src/views/DiagnosisView.vue` 第 66-110 行
- **问题描述**：`allDiagnoses` 对象硬编码了 3 个设备的诊断结论，其他设备走 `defaultDiagnosis` 兜底，生产环境无法维护
- **改进建议**：诊断数据应由后端 AI 接口返回

### L10. 性能：KnowledgeGraphView 案例数据硬编码且与 DiagnosisView 重复
- **文件**：`src/views/KnowledgeGraphView.vue` 第 88-200 行
- **文件**：`src/views/DiagnosisView.vue` 第 119-195 行
- **问题描述**：A引风机、A磨煤机的推理图数据在两个文件中各定义一份，内容高度重复但不完全一致（pct 值不同）
- **改进建议**：抽取到 `data/graphCases.js` 统一管理

### L11. 交互：ReasoningGraph 拖动节点坐标转换公式可能不准确
- **文件**：`src/components/ReasoningGraph.vue` 第 219-224 行
- **问题描述**：`px = (e.clientX * ctm.a + e.clientY * ctm.c + ctm.e)` 这个矩阵变换公式忽略了 transform: translate/scale 的影响（transform 是 CSS 层面，getScreenCTM 包含了 transform），但 SVG 内部坐标和 overrides 存储的坐标混用，缩放后拖动会偏移
- **改进建议**：拖动时考虑 scale 因子，或用 `svg.createSVGPoint().matrixTransform()`

### L12. 交互：表单未做"未保存提示"
- **文件**：`src/views/AlarmConfigView.vue` 第 334-337 行
- **问题描述**：`resetForm`（取消）直接清空表单，如果用户已修改未保存，无二次确认
- **改进建议**：增加"确认放弃修改？"弹窗

### L13. 一致性：报警等级颜色未集中管理
- **文件**：多处使用 `#ef4444`（一级）、`#f59e0b`（二级）、`#06b6d4`（智能预警）
- **问题描述**：颜色值在 10+ 文件中硬编码，若需调整需全局搜索替换
- **改进建议**：抽取为 SCSS 变量或 JS 常量 `theme.js`

### L14. 性能：AppLayout clock 每秒更新触发整个组件 re-render
- **文件**：`src/components/AppLayout.vue` 第 69-73 行
- **问题描述**：`clock` ref 每秒更新，导致 AppLayout 整个 re-render，包括 `<router-view />`（虽然 Vue 会 diff 跳过，但仍有开销）
- **改进建议**：将 clock 抽取为独立子组件

### L15. 交互：搜索/筛选能力薄弱
- **文件**：`src/views/AlarmListView.vue`
- **问题描述**：报警列表只能按等级筛选，不能按设备/专业/时间范围/状态组合筛选
- **改进建议**：增加多条件筛选表单

### L16. 交互：无快捷键支持
- **文件**：全局
- **问题描述**：工业 SCADA 系统通常需要快捷键（如 F1 帮助、Ctrl+K 全局搜索、Alt+1/2/3 切换页面），本项目仅 LoginView 密码框有 `@keyup.enter`
- **改进建议**：使用 `vueuse` 的 `useMagicKeys` 添加全局快捷键

### L17. 代码风格：scoped style 中 :deep() 滥用 !important
- **文件**：`src/assets/global.css` 多处
- **文件**：`AppLayout.vue` 第 125-130 行
- **问题描述**：大量 `!important` 覆盖 Element Plus 默认样式，未来升级 Element Plus 版本时容易冲突
- **改进建议**：使用 Element Plus 的 CSS 变量主题定制

### L18. 可访问性：ECharts 图表无 alt/aria 描述
- **文件**：所有 ECharts 容器
- **问题描述**：`<div ref="ch1" style="height:200px"></div>` 无 `role="img"` `aria-label`，屏幕阅读器无法识别图表内容
- **改进建议**：增加 `role="img" :aria-label="图表描述"`

### L19. 数据流：selectedDevice 全局状态被多个页面共享但无同步保护
- **文件**：`src/stores/data.js` 第 275 行
- **问题描述**：`selectedDevice = ref('U1-idf-a')` 是全局状态，DeviceManageView 修改后会影响 DiagnosisView/ConditionAnalysisView，但用户感知不到这种联动
- **改进建议**：各页面用本地 ref 管理选中状态，或显式提示"已同步到其他页面"

### L20. 交互：el-input-number 缺少单位提示
- **文件**：`src/views/DeviceManageView.vue` 第 47 行
- **问题描述**：`<el-input-number v-model="row.value" :min="0" :step="row.step" />` 用户不知道单位是 ℃ 还是 mm/s
- **改进建议**：在输入框后增加单位标识

## 四、按维度的总结

### 1. 板块展示逻辑
- **主要问题**：DashboardView 信息密度过高（机组选择+专业切换+概览+排放+健康度+3图表+AI洞察 同屏），工业现场大屏可读性差
- **主要问题**：GlobalAlertView 与 DashboardView 功能高度重叠（都有机组选择、KPI、设备健康度、AI洞察），定位不清
- **主要问题**：AlarmListView 的"操作指导"板块只有在点击操作后才显示（`v-if="cur"`），首次进入页面无引导
- **主要问题**：DiagnosisView 的引用溯源放在最底部，需滚动才能看到，但它是核心功能

### 2. 导航栏效果
- **主要问题**：菜单分组逻辑混乱（见 H11）
- **主要问题**：无面包屑（见 L6）

- **主要问题**：菜单选中态不响应式（见 L7）

### 3. 图片/图表对比度
- **主要问题**：图表文字过小（见 M21）
- **主要问题**：次要文字对比度不足（见 M20）
- **主要问题**：橙色与黄色辨识度低（见 L3）
- **亮点**：状态色（红/橙/绿）基本符合工业标准

### 4. 人机交互
- **主要问题**：批量操作无二次确认（见 H12）
- **主要问题**：无分页（见 M2）
- **主要问题**：无 loading/空状态（见 M3）
- **主要问题**：表单验证不完善（见 M4）
- **主要问题**：AgentView 无 loading（见 M11）
- **主要问题**：无快捷键（见 L16）

### 5. 其他维度
- **代码质量**：样式重复定义（M16）、函数重复（M14/M15）、硬编码数据（L9/L10）
- **性能**：Element Plus 全量引入（H3）、ECharts 未封装（H9）、setInterval 泄漏（H10）
- **可访问性**：全局缺失（M18/L18）
- **响应式**：除登录页外无适配（M19）
- **一致性**：命名风格混乱（L1）、颜色未集中管理（L13）
