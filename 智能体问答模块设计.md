# 智能体问答模块设计（AI Agent Chat）

> 对应系统：`运行智能体` 页面（`/agent`）及后端服务 `agent.service.url:9001`。
> 目标：把现有 `AgentView.vue` 中的本地 mock 对话，升级为“可调后端、可调 LLM、可调知识库”的真实智能体问答服务。

---

## 1. 设计目标

| 目标 | 说明 |
|------|------|
| 真实对话 | 用户输入 → LLM 推理 → 工具调用 → 生成文本/卡片/图表，不再写死 mock。 |
| 多轮会话 | 支持新建、切换、搜索历史会话，会话记录落库。 |
| 数据增强 | 可调用后端 API（设备、报警、统计）和算法服务 `/algo` 拿实时数据。 |
| 知识增强 | 可检索知识库 / 知识图谱，生成带依据的回答。 |
| 可扩展 | 后续接入新工具、新模型、新数据源，只需加函数和 prompt。 |

---

## 2. 总体架构

```
┌────────────────────────────────────────────────────────────────────┐
│                         用户浏览器（AgentView.vue）                    │
│  - 会话侧边栏 / 8 大功能模式 / 消息流 / 卡片 / 图表 / 快捷问题        │
└───────────────┬────────────────────────────────────────────────────┘
                │  HTTP / SSE  /api/agent/chat
                │  POST /api/agent/sessions
                │  GET  /api/agent/sessions/:id/messages
                ▼
┌────────────────────────────────────────────────────────────────────┐
│  Agent 服务（FastAPI，端口 9001，容器名 ppa-agent）                   │
│  - 会话管理：Session / Message / ToolCall 表                          │
│  - 流式推理：OpenAI-compatible LLM + SSE                               │
│  - 函数调用：Tools Dispatcher（查设备、查报警、查趋势、查知识库）       │
│  - 响应格式化：把 LLM 输出转成前端需要的 text / card / chart 结构      │
└───────────────┬────────────────────────────────────────────────────┘
                │
       ┌────────┴────────┬─────────────────┬─────────────────┐
       ▼                 ▼                 ▼                 ▼
  PostgreSQL        Spring Boot        算法 / KG         知识库
  ppa_db (agent_*)  backend:8080       /algo / kg:9000   文本/向量表
```

## 3. 数据模型（PostgreSQL）

复用现有 `ppa_db` 实例，新增 `agent_` 前缀表。`conversation_history` 表为 `jsonb[]`，便于后续重放和调试。

```sql
CREATE TABLE agent_sessions (
    id            BIGSERIAL PRIMARY KEY,
    title         VARCHAR(200) NOT NULL DEFAULT '新对话',
    mode          VARCHAR(50),                 -- 设备工况/报警分析/...
    unit_id       VARCHAR(50),                 -- 可选机组上下文
    created_by    VARCHAR(100) NOT NULL,
    created_at    TIMESTAMPTZ DEFAULT NOW(),
    updated_at    TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE agent_messages (
    id            BIGSERIAL PRIMARY KEY,
    session_id    BIGINT NOT NULL REFERENCES agent_sessions(id) ON DELETE CASCADE,
    role          VARCHAR(20) NOT NULL CHECK (role IN ('user','assistant','tool','system')),
    content       TEXT,
    -- 结构化内容（LLM 最终输出）
    payload       JSONB DEFAULT '{}',          -- {text, card, chart, quick_questions, templates}
    -- LLM 原始推理过程（可折叠）
    thinking      TEXT,
    -- 工具调用记录
    tool_calls    JSONB DEFAULT '[]',        -- [{name, args, result, latency_ms}]
    -- 模型与 token 统计
    model         VARCHAR(100),
    prompt_tokens INT,
    completion_tokens INT,
    created_at    TIMESTAMPTZ DEFAULT NOW(),
    INDEX idx_session_created (session_id, created_at)
);

CREATE TABLE agent_knowledge (
    id            BIGSERIAL PRIMARY KEY,
    category      VARCHAR(50),                 -- 知识库/政策/运行规程/故障案例
    title         VARCHAR(300),
    content       TEXT NOT NULL,
    tags          TEXT[],
    source        VARCHAR(300),                -- 来源文件或规范编号
    ts_vector     TSVECTOR,                    -- PostgreSQL 全文检索
    created_at    TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_agent_knowledge_ts ON agent_knowledge USING GIN(ts_vector);
CREATE INDEX idx_agent_knowledge_cat ON agent_knowledge(category);
```

## 4. 后端 API 设计（FastAPI）

统一前缀 `/api`（与项目约定一致）。流式接口用 `text/event-stream`。

### 4.1 会话管理

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/agent/sessions` | 分页查会话列表（含最新消息摘要） |
| POST | `/api/agent/sessions` | 新建会话 `{title, mode, unit_id, created_by}` |
| GET | `/api/agent/sessions/{id}/messages` | 取会话历史消息 |
| DELETE | `/api/agent/sessions/{id}` | 删除会话 |
| PUT | `/api/agent/sessions/{id}/title` | 重命名会话 |

### 4.2 智能问答（流式）

```
POST /api/agent/chat/stream
Content-Type: application/json
```

请求体：

```json
{
  "session_id": 100,
  "question": "A引风机为什么健康度低？",
  "mode": "故障溯源",
  "unit_id": "U1",
  "stream": true
}
```

SSE 事件流：

```text
event: thinking
data: {"type":"thinking","content":"正在分析A引风机历史工况..."}

event: tool_call
data: {"type":"tool_call","name":"query_device_status","args":{"device":"A引风机"}}

event: tool_result
data: {"type":"tool_result","name":"query_device_status","result":{...}}

event: content
data: {"type":"content","delta":"A引风机当前轴承温度 82.3℃，已超出 75℃ 阈值。"}

event: card
data: {"type":"card","card":{"title":"A引风机 · 健康度 82.3","stat":"温度超限","body":...}}

event: chart
data: {"type":"chart","chart":{"title":"A引风机趋势","source":"Trend","id":"t1_xxx"}}

event: done
data: {"type":"done","message_id":1234}
```

非流式版本：`POST /api/agent/chat` 直接返回完整 JSON 响应。

## 5. LLM 与 RAG 流程

### 5.1 推理流程

```
用户问题
  │
  ▼
组装 system prompt（身份 + 工具描述 + 输出格式）
  │
  ▼
检索知识库（可选 top-k）
  │
  ▼
调用 LLM (OpenAI-compatible /tools)
  │
  ▼
解析函数调用 → 执行 Tool Dispatcher
  │
  ▼
将工具结果再次喂给 LLM → 生成最终回答
  │
  ▼
结构化输出：text / card / chart / quick_questions / templates
```

### 5.2 System Prompt 核心规则

```
你是“火电厂智能运行助手”，熟悉火力发电设备、运行规程、报警处置。
当前模式：{mode}。机组上下文：{unit_id}。

可使用的工具：{tool_descriptions}

输出规则：
1. 如果回答涉及设备状态，请用 card 字段输出状态卡片。
2. 如果需要趋势/工况图，请用 chart 字段输出图表配置。
3. 若问题可追问，在 quick_questions 中给出 3 条推荐问题。
4. 思考过程请放在 thinking 字段，但面向用户的结论要简洁。
```

### 5.3 知识库检索（短期方案）

- 使用 PostgreSQL `ts_vector` 全文检索，按 `ts_rank` 排序。
- 提供 `/api/agent/knowledge/search?q=引风机失速`。
- 命中结果拼接进 prompt 上下文。

> 长期：接入独立 `kg.service.url:9000` 做图谱检索。

## 6. 工具 / 函数调用设计

每个工具返回标准 JSON，Agent 负责解释并生成卡片/图表。

| 工具名 | 作用 | 参数 | 输出 |
|--------|------|------|------|
| `query_device_status` | 查询设备实时状态 | `device_id` | 健康度、负荷、振动、温度等 |
| `query_alarms` | 查询报警 | `level`, `status`, `time_range` | 报警列表 |
| `query_trend` | 查趋势 | `metric`, `device_id`, `time_range` | 时间序列数据 |
| `query_knowledge` | 知识库检索 | `q` | 相关知识片段 |
| `query_working_condition` | 查工况图 | `device_id` | 工况曲线数据 |
| `generate_report` | 生成运行日志/报告 | `report_type` | Markdown / 结构化文本 |

**工具执行层**：通过 `httpx` 调用 `http://backend:8080`（Docker 内）或 `http://localhost:8080`（本地），复用后端已有 `/api/**` 接口。

## 7. 前端接入设计

### 7.1 现有 `AgentView.vue` 改动点

| 位置 | 现状 | 改动 |
|------|------|------|
| `chatHistory` | 本地写死数组 | 初始化时 `agentApi.sessions()` 拉取 |
| `newChat` | 本地 push | 调用 `agentApi.createSession()` 后更新列表 |
| `openChat` | 本地切换 | 调用 `agentApi.messages(id)` 加载历史 |
| `send` | 本地 setTimeout 生成 mock | 通过 SSE 连接后端，实时写入消息 |
| `generateCard / generateChart` | 本地硬编码 | 接收后端 `card` / `chart` payload 并渲染 |

### 7.2 API 客户端（`frontend/src/api/agent.js`）扩展

```js
export const agentApi = {
  sessions: (page = 1, size = 20) => request.get('/agent/sessions', { params: { page, size } }),
  createSession: (data) => request.post('/agent/sessions', data),
  messages: (id) => request.get(`/agent/sessions/${id}/messages`),
  rename: (id, title) => request.put(`/agent/sessions/${id}/title`, { title }),
  delete: (id) => request.delete(`/agent/sessions/${id}`),
  chatStream: (data) => new EventSource(`/api/agent/chat/stream?${new URLSearchParams(data)}`)
}
```

### 7.3 响应渲染映射

| 后端 payload 字段 | 前端渲染 |
|-------------------|----------|
| `text` | `.ai-text` |
| `card` | `.ai-card`（含 title / stat / body / ok） |
| `chart` | `.ai-chart`（用 ECharts 渲染） |
| `thinking` | `.ai-thinking`（可折叠） |
| `quick_questions` | `.ai-quick-q` 推荐问题 |
| `templates` | `.ai-templates` 快捷模板 |

## 8. 部署与集成

### 8.1 在 `docker-compose.yml` 新增 `agent` 服务

```yaml
agent:
  build:
    context: ./agent
    dockerfile: Dockerfile
  container_name: ppa-agent
  restart: unless-stopped
  environment:
    TZ: Asia/Shanghai
    DATABASE_URL: postgresql://ppa_user:ppa_pass_2024@postgres:5432/ppa_db
    REDIS_URL: redis://redis:6379/0
    BACKEND_URL: http://backend:8080
    LLM_BASE_URL: ${LLM_BASE_URL:-https://api.openai.com/v1}
    LLM_API_KEY: ${LLM_API_KEY:-}
    LLM_MODEL: ${LLM_MODEL:-gpt-4o-mini}
  ports:
    - "9001:9001"
  depends_on:
    postgres:
      condition: service_healthy
    backend:
      condition: service_healthy
```

### 8.2 与主后端的区别

- `backend:8080`：Spring Boot 主后端，处理业务数据。
- `agent:9001`：FastAPI 智能体服务，处理对话推理、LLM、工具编排。
- 前端 `AgentView.vue` 直接对话 `agent:9001`，但 `agent` 内部会通过 `BACKEND_URL` 调用主后端拿数据。

### 8.3 目录结构

```
ppa-integrated/
├── agent/                    # 新增智能体服务
│   ├── Dockerfile
│   ├── requirements.txt      # fastapi / uvicorn / sqlalchemy / httpx / openai / psycopg2-binary
│   ├── app/
│   │   ├── main.py           # FastAPI 入口
│   │   ├── models.py         # SQLAlchemy 模型
│   │   ├── schemas.py        # Pydantic 请求/响应
│   │   ├── chat.py           # LLM 推理 + SSE 流
│   │   ├── tools.py          # 工具定义与 dispatcher
│   │   ├── repository.py     # 会话/消息 CRUD
│   │   └── knowledge.py      # 知识库检索
│   └── init.sql              # agent_sessions / agent_messages / agent_knowledge
```

## 9. 实施计划（建议顺序）

1. **搭建骨架**：创建 `agent/` 目录，Dockerfile，FastAPI 入口，health check。  
2. **数据层**：`models.py` + `init.sql`，实现会话/消息 CRUD API。  
3. **接入前端**：`AgentView.vue` 从后端拉取会话/历史，消息展示保持现有结构。  
4. **LLM 基础对话**：流式 `POST /api/agent/chat/stream`，先只返回 `text`。  
5. **工具调用**：实现 `query_device_status`、`query_alarms`、`query_trend` 等工具。  
6. **结构化输出**：让 LLM 输出 `card` / `chart` payload，匹配截图效果。  
7. **知识库**：导入文档、建全文索引，实现 `query_knowledge`。  
8. **Docker 编排**：把 `agent` 加入 `docker-compose.yml`，与 `nginx` 联动。  

---

## 10. 关键约定

- **消息格式**：后端统一用 SSE 事件，`payload` 字段决定前端渲染形式；前端保留现有卡片/图表样式。  
- **会话标题**：首次用户提问后，用 LLM 生成 8 字以内标题，自动更新 `session.title`。  
- **知识库**：先轻量 PostgreSQL 全文检索，后迁移至 `kg.service.url:9000` 图谱检索。  
- **模型配置**：优先使用 OpenAI-compatible API，可一键切换阿里云、DeepSeek、本地 vLLM/Ollama。  
- **安全**：LLM 配置项（`LLM_API_KEY`）走 `.env`，不提交代码。
